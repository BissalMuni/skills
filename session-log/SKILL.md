# 세션 로그 — 프로젝트 성장·디버깅 일지

현재 Claude Code 대화의 핵심 내용을 **현재 작업 중인 프로젝트**의 `docs/sessions/` 폴더에 날짜시간 또는 버전 기반 Markdown 파일로 저장한다. 단순 요약이 아니라 **프로젝트가 어떻게 성장했는지, 어떤 디버깅을 거쳤는지 추적 가능한 작업 일지**를 만드는 것이 목표.

## 언제 실행하나

- 사용자가 명시적으로 요청: `/session-log`, "세션 기록", "오늘 작업 정리", "이거 문서로 남겨줘"
- 중요한 마일스톤 직후 (기능 구현 완료, 큰 버그 해결, 설계 전환 등)
- 세션 종료 직전 사용자가 작업 내용 보존을 원할 때

## 입력

```text
$ARGUMENTS
```

인자로 받을 수 있는 것:
- **파일명 힌트**: `v0.8.0` / `2026-04-21` / `refactor-auth` 등
- **제목**: "온톨로지 평가 시스템 설계"
- **범위 제한**: "오늘 작업만", "최근 1시간", "버그 수정 부분만"

인자가 없으면 자동으로 판단한다.

## Workflow

### Step 1: 대화 분석

현재 컨텍스트에서 다음 요소를 추출한다:

1. **세션 목표**: 사용자가 이번 대화에서 달성하려 한 것
2. **주요 작업 흐름**: 시간순으로 벌어진 일
   - 조사·분석 (파일 읽기, 그래프 질의, API 호출 등)
   - 설계 결정 (아키텍처 선택, 대안 비교)
   - 구현 (파일 생성·수정, 실행·검증)
   - 디버깅 (오류 발견 → 원인 분석 → 해결)
3. **핵심 결정과 근거**: "왜 A가 아니라 B를 선택했는가"
4. **코드·파일 변경**: 어떤 파일을 만들었고·수정했고·삭제했는가
5. **발견된 버그/한계**: 해결됐거나 남아있는 문제
6. **후속 과제**: 오늘은 안 했지만 해야 할 일
7. **학습/인사이트**: 프로젝트 설계에 영향을 줄 관찰

### Step 2: 프로젝트 루트 확인 및 저장 경로 결정

1. **현재 프로젝트 루트 확인**:
   - 환경 변수나 대화에서 확인한 작업 디렉터리 사용
   - Git 저장소 루트가 있으면 그것을 기준으로
2. **저장 경로**: `{project_root}/docs/sessions/`
3. 폴더가 없으면 생성

### Step 3: 파일명 결정

두 가지 명명 방식 중 선택:

#### 방식 A — 날짜시간 기반 (기본값)
```
YYYY-MM-DD-HHMM-<slug>.md
```
예: `2026-04-21-2345-evaluation-system-design.md`

- 날짜·시간은 현재 시스템 시간 사용
- `<slug>`: 세션의 주요 주제를 영문 kebab-case 3~5단어로
  - 한국어 주제는 영문으로 의역 (예: "온톨로지 평가 시스템" → `ontology-evaluation-system`)

#### 방식 B — 버전 기반
사용자가 인자에 `v0.8.0` 같은 버전을 지정하거나, 프로젝트에 `package.json`·`pyproject.toml` 버전이 명시돼 있고 세션이 해당 버전 작업을 마무리하는 경우.
```
v<version>-<slug>.md
```
예: `v0.8.0-concept-expansion.md`

#### 선택 규칙
- 사용자 인자에 버전이 있으면 방식 B
- 사용자 인자에 날짜만 있으면 방식 A (그 날짜 사용)
- 둘 다 없으면 방식 A (현재 시각 자동)

### Step 4: 문서 작성

다음 템플릿을 따르되, **실제 작업 내용에 맞게 섹션을 취사선택**한다 (억지로 빈 섹션을 만들지 않음).

```markdown
---
date: YYYY-MM-DD
time: HH:MM
session_id: YYYY-MM-DD-HHMM
version: v0.8.0      # 방식 B일 때만
title: "세션 제목 (한글)"
slug: "english-kebab-case"
tags: [도메인, 작업유형, 사용기술]
files_changed:
  - path: relative/path/to/file
    change: created | modified | deleted
    summary: 한 줄 요약
related_sessions:
  - 2026-04-20-1800-previous-topic  # 있으면
---

# 세션 제목

## 🎯 목표
이번 세션에서 달성하려 한 것. 1~3문장.

## 📖 배경
왜 이 작업이 필요했는가. 직전 상태·발단. 선행 세션이 있으면 링크.

## 🔍 진행 과정

### 1. 조사·분석
- 어떤 파일·그래프를 확인했는지
- 주요 발견 (문제의 진짜 원인 등)

### 2. 설계·결정
- 비교한 대안
- 선택한 방안과 근거

### 3. 구현
- 수정·생성한 파일과 핵심 변경
- 주요 코드 스니펫 (필요 시)

### 4. 검증
- 어떻게 테스트했는지
- 결과 (Before → After)

## 🐛 디버깅 기록 (있을 때만)

### 문제 N: {증상}
- **원인**: 원인 분석
- **해결**: 적용한 수정
- **교훈**: 같은 패턴 재발 방지를 위한 메모

## 🔑 핵심 결정

| 결정 | 선택안 | 기각안 | 이유 |
|---|---|---|---|
| ... | ... | ... | ... |

## 📂 변경된 파일

- `path/to/file.py` — 한 줄 요약
- `path/to/another.md` — 한 줄 요약

## ✅ 완료된 것 / 🚧 남은 것

### 완료
- [x] 항목 1
- [x] 항목 2

### 후속 과제
- [ ] 항목 3
- [ ] 항목 4

## 💡 인사이트

나중에 설계·결정에 영향을 줄 관찰. 선택사항.

## 🔗 관련 문서

```

## 작성 규칙

### 내용 원칙
1. **시간 순서 존중**: 대화가 흘러간 순서대로 기록. 결과만 적는 게 아니라 **어떻게 거기 도달했는지** 궤적을 남긴다.
2. **"왜"를 담는다**: 단순히 "A를 했다"가 아니라 "B 대신 A를 했고 그 이유는 C"까지.
3. **실패·막힌 지점 포함**: 첫 시도가 실패했으면 그 실패도 기록. 미래의 같은 실수를 막는 데 유용하다.
4. **구체적 수치·파일 경로**: "어떤 파일", "몇 건", "어떤 조문" — 애매하게 "여러 개"라고 쓰지 않는다.
5. **대화 그대로 옮기지 말 것**: 요약·정제해서 6개월 뒤 읽어도 빠르게 맥락 파악 가능하게.

### 형식 원칙
1. **한국어 본문 + 영문 기술 용어**: 설명은 한국어, 변수명·라이브러리명·명령어는 영문 그대로.
2. **코드 블록에 언어 태그**: ```python, ```cypher, ```typescript 등.
3. **파일 경로는 프로젝트 루트 기준 상대경로**: `knowledge_graph/domain_schema.py` 형태.
4. **변경 라인 번호 포함 가능**: 중요한 부분은 `파일명:123-140` 형태로.
5. **다이어그램은 ASCII 박스**: Mermaid 의존하지 말고 모노스페이스로 바로 보이게.

### 길이
- 일반 세션: 150~400줄 권장
- 아주 큰 세션 (설계 + 구현 + 디버깅 모두): 600줄까지 허용
- 이보다 길어지면 **세션을 여러 문서로 분할**: `-part1`, `-part2` 또는 주제별로 나눔

### 금지 사항
- 비밀번호·API 키·토큰 같은 민감정보 포함 금지 (마스킹하거나 언급 회피)
- `git commit`이나 `git push` 자동 실행 금지 (세션 로그는 사용자가 원할 때 수동 커밋)
- 기존 세션 문서 덮어쓰기 금지 (파일명 충돌 시 접미사 `-2`, `-3` 추가)

## 특수 케이스

### 케이스 1: 세션이 실패로 끝났을 때
완성된 기능이 없어도 기록 가치가 있다. 대신 제목에 `wip-` 또는 `exploration-` 접두사를 붙인다.

```yaml
title: "이혼 재산분할 특례 조사 (미완)"
slug: "wip-divorce-property-exception"
```

본문에 **막힌 지점·가설·다음 시도 아이디어**를 명시적으로 남긴다.

### 케이스 2: 세션이 너무 짧을 때 (10분 미만, 변경 없음)
- 단순 질문-답변만 있고 프로젝트 상태에 변화가 없으면 **저장하지 않는다**.
- 대신 사용자에게 "이번 세션은 대화만 있었고 프로젝트 변경이 없어 로그를 건너뜁니다"라고 알린다.

### 케이스 3: 이미 같은 주제의 최근 세션이 있을 때
`docs/sessions/`에서 같은 주제/slug의 24시간 이내 세션을 확인. 있으면:
- 사용자에게 "같은 주제 어제 세션이 있는데, 새 문서로 만들까요 / 이어서 쓸까요?" 확인
- 이어쓰기 선택 시 기존 문서 하단에 `## 🔄 후속 (YYYY-MM-DD HH:MM)` 섹션 추가

### 케이스 4: 여러 프로젝트를 동시에 다룬 세션
각 프로젝트의 `docs/sessions/`에 **해당 프로젝트 관련 부분만** 분리해 저장한다. 공통 부분은 상호 링크로 참조.

### 케이스 5: 프로젝트에 `docs/` 폴더가 없는 경우
`docs/sessions/`를 생성한다. 이때 간단한 안내 `docs/sessions/README.md`도 함께 만든다:

```markdown
# 세션 로그

이 폴더에는 Claude Code와 함께 진행한 작업 세션 기록이 저장됩니다.
각 파일은 날짜시간 또는 버전 기반으로 명명되며, 프로젝트의 성장과 디버깅 과정을 추적합니다.

인덱스: [INDEX.md](./INDEX.md)

세션 로그 생성: `/session-log` 또는 "세션 기록해줘"
```

## 사용자와의 상호작용

### 확인할 것 (필요 시)
- 저장 위치(기본: 현재 프로젝트 `docs/sessions/`) 확인이 필요한 경우만
- 파일명 방식 (날짜 vs 버전) — 모호할 때만
- 포함 범위 — "전체 대화" vs "방금 작업한 부분만"

### 확인하지 않을 것
- 제목·태그·slug — 자동 생성하되, 사용자 인자 우선
- 본문 구조 — 위 템플릿에 맞춰 자동 작성

## Guidelines

- **정확성 > 속도**: 작성 전에 변경 파일·결정 근거를 다시 읽어 맞는지 확인
- **미래의 자신을 위한 글쓰기**: 3개월 뒤 이 문서만 보고 당시 맥락을 재구성할 수 있어야 함
- **연속성 유지**: 이전 세션 로그를 간단히 훑어 맥락 이어지게
- **Git 커밋은 사용자가 결정**: 스킬은 파일 생성까지만. 커밋은 사용자가 명시적으로 요청할 때만
- **민감정보 자동 스캔**: `API_KEY`, `PASSWORD`, `SECRET`, `TOKEN` 패턴이 대화에 나왔으면 본문에서 마스킹(`[REDACTED]`) 후 저장
